Laboratory System Central Database - Version 2.7

Updated 26 September 2013

Most tables include a unique record ID, taken from c_id_sequence. Tables and fields marked as obsolete should not be used by new programs; they will be removed as the roll-out of v2.7 continues. Obsolete fields and those added to existing tables have default values so that old and new programs alike can safely add data.

New programs should access the labs database using the (64-bit) vNode 'vnode_lab_64'. Existing code uses the (32-bit) ODBC connection 'dsn_lab_ingres' or 'dsn_lab_star'. Old programs also use a BDE alias, 'alias_local_buddy', for temporary storage. This may point to a MySQL database on the cluster controller or it may point to Ingres on vlab. A similar setup exists on the laptop PC(s) used for the storage sub-system, using Ingres locally.

BUDDY_DATABASE

A record is added to this table when a list of assays is sent to an analyser. Comms links results to it as they are returned from the analyser. Results from a single run may occasionally appear to be from multiple analyses, in which case there will be multiple buddy_database records. Validation will link these using sample_run_id

Name Type Key Reference Description
buddy_sample_id i4 1 A unique ID for this analysis
machine_cid i4 c_machine The analyser the assay list was sent to and/or the results were produced by
barcode v32 The barcode read by the analyser
date_analysed d/t When the assay list was sent to the analyser (if programmed), or when the results were received
action_flag v1 '7': project/sample ID not known; '8': sample programmed; '0': results awaiting validation; '9': results wiped from analyser; '1': results validated at first level; '6': keep for later; '4': all results invalidated; '2': validation complete (use with caution: buddy_result_float's action_flag is more specific); '3': results sent to project; 'X': closed off
database_name v20 c_project The project from which the sample came or the database to which results have been sent ("unknown" or empty if no work list entries)
alpha_sample_id i4 specimen The (project-specific) ID of the specimen this came from (0 if no work list entries)
rack_number v16 The barcode of the rack or plate this sample came from (if known, otherwise blank, '.' or '0')
cup_number i2 The position of the sample in the rack (if known, otherwise 0)
sample_type v20 c_test_machine The analyser's code for the sample type, e.g. "Whole Blood" (DXH) or "U" (AU680)
update_when d/t When this record was most recently modified - used by NightShift to determine when it is safe to delete the record
note_exists i2 The number of notes for this analysis (in buddy_note, linked on buddy_sample_id)

C_BOX_RETRIEVAL

Retrieval plans, created and used by StoreMan's Retrieval Assistant. Sample retrieval/disposal tasks will also have entries in l_cryovial_retrieval.

Each retrieval may be divided up into sections/chunks to be carried out in a particular order. The plan for a box retrieval/disposal is given by
select * from c_box_retrieval b where retrieval_cid = ? order by b.section, b.rj_box_cid

retrieval_type i2 obsolete - see c_retrieval_job
box_id i4 box_name The box being retrieved (for box retrieval/disposal) or retrieved into (for sample retrieval/disposal)
section i2 Which chunk of the retrieval plan this entry belongs to (0 = retrieve all boxes in parallel)
position i2 obsolete
box_name v32 obsolete
rj_box_cid i4 1 Unique ID for this retrieval list entry (also determines retrieval order for box retrievals)
status i2 0: new record; 1: part-filled, 2: collected; 3: not found; 99: record deleted
time_stamp d/t When this record was inserted or updated

C_TEST_DEFINITION

A list of calculated tests/checks. Results are calculated by working through records with the same result_test_cid:

  1. start with a subtotal of 0
  2. for each record, in test_position order:
    1. find the next not-yet-validated result for source_test_cid (use 1.0 if source_test_cid=0)
    2. multiply that result by source_multiple
    3. take the previous subtotal and add, subtract, multiply or divide by the product of step 2
  3. store the total as the result for result_test_cid
Field name Type Key Refers to Description / Comments
result_test_cid i4 c_test The value being calculated
calculation i2 The calculation to be performed for this step: 0 = subtract, 1 = divide, 2 = add, 3 = multiply
test_position i2 The position of this step in the calculation
source_test_cid i4 c_test The result to be used for this step (0: use source_multiple as a constant)
source_multiple f8 Multiply the source result by this factor
valid_from d/t When this (part of the) definition changed
valid_to d/t When this (part of the) definition will no longer apply
record_cid i4 1 unique id for this record
status i2 current status: 0 = new record, 1 = in use, 99 = obsolete
