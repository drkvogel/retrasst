Laboratory System Project Databases - Version 2.7

Updated 27 September 2013

New programs should access the labs database using the Ingres x-library using the (64-bit) vNode 'vnode_lab_64'. Existing code uses the (32-bit) ODBC connection 'dsn_lab_ingres' or 'dsn_lab_star'. Fields and tables marked as obsolete should not be used - they may be deleted in future versions of the database.

Most tables include a unique record ID, taken from id_sequence

Sample Reception identify the type of box, cryovial or vacutainer using a combination of "by-hand" descriptor values, then find a matching specimen record using the barcode. There should not be more than one matching record. The descriptor values in the corresponding specimen record can then be used to identify a test/aliquot profile - there should be one record in test_profile_description that matches (a sub-set of) the record's values.

By-hand descriptors might include:

  • analysis (required): the type of procedure requested, e.g. 'storage'
  • vial_size: 'C' = cryovial, 'L' = large vacutainer etc.
  • tube_type: 'EDTA', Urine etc. (should match an analysis category in c_object_name)
  • aliquot_type (required): each value should match an aliquot type in c_object_name

Other descriptors might include:

  • barcode or box_label - the tube or box the sample(s) arrived in (specimen_field_name = 'barcode'; descriptor_flags = 13)
  • source_name (required) - links entries for samples from a single participant visit, including any aliquots taken from those samples
  • first_cryovial and last_cryovial - the barcodes of the first and last cryovials in a box (used by Box Reception)
  • long_term_storage - has participant consented

SPECIMEN

An entry for each occasion a box, kit, vacutainer or cryovial needs to be processed by the labs. Projects can add records to this table to notify LIMS when something is expected. The operator scans a vacutainer or cryovial into Specimen Reception which checks the descriptor values and creates appropriate worklist entries. Later, results or cryovials are attached. Boxes are matched through Box Reception

Name Type Key Reference Description
sample_id i4 1 A project-unique ID for this record
parent_id i4 specimen The sample_id of the source this is taken from, otherwise 0
external_name v32 A table-unique name for this entry. For a kit, this should match the source name. For a box, this may be the name written on the box
barcode v32 The barcode on the box, cryovial or vacutainer. For a box without a barcode, this may be the name written on the box
status i2 0: expected sample, 1: received, 2: results available, 3: results and/or cryovial data collected, 4: cryovials stored, 5: unexpected (project notified), 6: discarded, 7: source/kit details, 8: expected box 99: obsolete/invalid
specimen_taken d/t when the blood/urine was taken (usually the appointment date, if known; a.k.a. "in" date )
swipe_time d/t when the tube was swiped in by labs staff (set by Sample Reception or Box Reception when status changes to 1)
update_when d/t when this record was last updated, if it has been
insert_who v12 The person or program that first made this entry
insert_when d/t when this record was added to the table ( default: 'Now' )
process_cid i4 c_audit_trail The run of the program that created or updated this record
operator_cid i4 c_operator The operator that swiped this sample in
profile_id i4 test_profile_description the ID of the identified profile (can be updated when sample is swiped in)
note_exists i2 whether a note has been added (0: no, >=1: yes)
value1 v12 the value for the descriptor with a specimen_field_name of value1 [default .]
value2 v12 the value for the descriptor with a specimen_field_name of value2 [default .]
value3 v12 the value for the descriptor with a specimen_field_name of value3 [default .]
etc. up to value25

TEST_PROFILE_DESCRIPTION

Gives a name to a type of sample and what to do when it arrives

Name Type Key Reference Description
test_profile_id i4 1 an ID for this test and/or aliquot profile
test_profile_name v32 A project-unique name of this profile
test_profile_description v128 a description of this test/aliquot profile
date_last_modified d/t when this description was last changed
note_exists i2 how many notes regarding this record have been entered
profile_flags i4 0: disposal (no tests or aliquots added to the worklists; no aliquot_profile with status<> 99 or test_profile records with test_flags<>99)
1: take aliquots - Specimen Reception creates records in c_box_worklist with status 'Q' (for entries in aliquot_profile with status 0)
2: analyse the sample - Specimen Reception creates entries in c_buddy_worklist with status 'Q' (for entries in test_profile with test_flags<>99)
3: analyse the sample and store aliquots from it - Specimen Reception creates entries in c_buddy_worklist and c_box_worklist
4: store the original sample (SR creates a record in c_box_worklist with 'P' status for entries in aliquot_profile with status 1)
5: take aliquots and store original (SR adds c_box_worklist records with status 'Q' for aliquot_profile status 0, 'P' for status 1)
6: analyse and store the original (c_buddy_worklist 'Q' from test_profile, c_box_worklist 'P' from aliquot_profile status 1)
7: analyse, take aliquots and store original (c_buddy_worklist 'Q' from test_profile, c_box_worklist 'P' or 'Q' from aliquot_profile 1 or 0)
status i2 0: new record; 1: in used/modified; 99: obsolete/ignored
process_cid i4 c_audit_trail The program that created/updated this record
